# 手動テストチェックリスト

Issues #13–#28 の UI 機能を対象とした手動テスト手順。

---

## 前提：アプリ起動 & フォルダを開く

- [ ] `pnpm tauri dev` でアプリ起動
- [ ] 「ワーキングフォルダを開く」で PDF が入ったフォルダを選択
- [ ] 中央ペインに PDF ファイル一覧が表示される

---

## テスト 1 — 変更種別バッジの表示（#21）

**前提：** PDF を1つ編集するか、新規で追加する

- [ ] CommitPanel（右ペイン上部）に「変更あり」と保留ファイルが表示される
- [ ] 新規ファイル → `M` バッジが付く
- [ ] 変更ファイル（バイト差異あり）→ `M` または `~` バッジが付く
- [ ] 初回コミット前（git 履歴なし）の状態でも保留一覧が出る

---

## テスト 2 — `~` バッジを M に昇格（#22）

**前提：** `~` バッジが付いているファイルがある状態

- [ ] `~` バッジをクリック → `M` に変わる
- [ ] 変わった後の `M` をクリック → 何も変化しない（ダウングレード不可）

---

## テスト 3 — コミット & 保留リストのクリア（#14）

- [ ] CommitPanel のテキストエリアにメッセージを入力して「コミット」ボタンを押す
- [ ] 保留変更リストが消える（パネルごと非表示になる）
- [ ] CommitHistoryPanel に新しい履歴が追加される

---

## テスト 4 — 図面クリック → コミット履歴表示（#18）

- [ ] 中央ペインで図面ファイル名をクリック
- [ ] CommitHistoryPanel にそのファイルのコミット履歴（メッセージ・作者・日時）が表示される
- [ ] まだ一度もコミットしていないファイルをクリック → 「コミット履歴がありません」と表示される

---

## テスト 5 — 作業コピー差分表示（#21）

**前提：** 1回以上コミット済みで、さらに同じファイルを修正した状態

- [ ] CommitPanel でその修正ファイルをクリック
- [ ] CommitPanel 内の DiffView に差分オーバーレイ画像が表示される
  - 変更量が大きい場合：赤（削除）・青（追加）のオーバーレイ
  - 変更がほぼない場合：画像なし

---

## テスト 6 — コミット履歴で2点選択 → 差分表示（#20）

**前提：** 同じファイルに対して2つ以上のコミットがある状態

- [ ] CommitHistoryPanel で古いコミット（旧版）をクリック → ハイライトされる
- [ ] 別のコミット（新版）をクリック → DiffView に差分オーバーレイが表示される
- [ ] 「変更なし」「微小変更」「意味的変更」のいずれかのラベルが表示される
- [ ] 同じコミットを2回選択した場合 → 「変更なし」になる

---

## テスト 7 — ファイル変更の自動検知（#15/#16）

- [ ] ワーキングフォルダを開いた状態で、アプリの外から PDF を追加または更新
- [ ] CommitPanel が自動更新され、保留ファイルとして表示される（手動リロード不要）
- [ ] 画面下部にポーリングモードのバナーが表示されている場合はネイティブ検知が無効（正常）

---

## テスト 8 — 初回起動時のユーザー名入力（#24）

**前提：** `velgit-store` を localStorage から削除してアプリを再起動（DevTools → Application → Local Storage → 削除）

- [ ] アプリ起動直後にユーザー名入力モーダルが表示される
- [ ] メイン画面の操作が一切できない（モーダルの背後はブロックされている）
- [ ] 入力フィールドが空の状態では「決定」ボタンが非活性になっている
- [ ] 名前を入力して Enter キーを押す → モーダルが閉じてメイン画面に遷移する
- [ ] 名前を入力して「決定」ボタンをクリック → 同様にメイン画面に遷移する

---

## テスト 9 — ユーザー名の永続化と変更（#24）

**前提：** テスト 8 でユーザー名を設定済み

- [ ] アプリを再起動してもユーザー名入力モーダルが表示されない（設定が保持されている）
- [ ] 左ペイン上部に現在のユーザー名が表示されている
- [ ] ユーザー名をクリック → 編集フィールドに切り替わる
- [ ] 新しい名前を入力して「保存」ボタンをクリック → 表示名が更新される
- [ ] 編集中に Escape キーを押す → 変更せずに元の表示に戻る

---

## テスト 10 — コミット作者名の記録（#24）

**前提：** ユーザー名を設定済みで、PDF をコミットできる状態

- [ ] PDF を追加・変更してコミットを実行する
- [ ] CommitHistoryPanel の「作者」欄に設定したユーザー名が表示される
- [ ] ユーザー名を変更して再度コミット → 新しいユーザー名が作者として記録される

---

## テスト 11 — 図渡し作成フォーム（#26）

**前提：** ワーキングフォルダを開いて図面が1件以上ある状態、かつ1回以上コミット済み

- [ ] 右ペインの最下部に「図渡し作成」フォームが表示される
- [ ] 図面チェックリストに全図面が表示され、すべてチェック済みになっている
- [ ] 図渡し名称を空のまま → 「図渡しを作成」ボタンが非活性
- [ ] すべての図面のチェックを外す → ボタンが非活性のまま
- [ ] 名称を入力し、図面を1件以上選択 → ボタンが活性化する

---

## テスト 12 — 図渡し送信の動作（#26）

**前提：** テスト 11 の条件を満たしている状態

- [ ] 種別ラジオのデフォルトが「社内図渡し」になっている
- [ ] 「社外図渡し」を選択できる
- [ ] 相手先フィールドを空のまま送信できる（エラーにならない）
- [ ] 図面を一部だけ選択して送信できる
- [ ] 「図渡しを作成」をクリックすると送信中はボタンが「送信中...」表示になる
- [ ] 送信成功後、名称フィールドがクリアされ、図面が全選択に戻る

---

## テスト 13 — バインドPDF生成（#28）

**前提：** 図渡しが1件以上あり、かつ含まれる図面が HEAD にコミット済みの状態

- [ ] 図渡し履歴の各エントリに「バインドPDF生成」ボタンが表示される
- [ ] ボタンをクリックするとOS の保存先選択ダイアログが開く
- [ ] 保存先を選択すると「生成中...」表示に切り替わる
- [ ] 生成完了後、指定したパスに PDF ファイルが書き出される
- [ ] 書き出された PDF を外部ビューアで開くと、図渡しに含まれる全図面が結合されている
- [ ] ページ順序がファイル名昇順になっている（例: A-001.pdf → S-001.pdf の順）
- [ ] HEAD に存在しない図面が含まれている場合、エラーメッセージが表示される
- [ ] 保存ダイアログでキャンセルすると何も起きない
