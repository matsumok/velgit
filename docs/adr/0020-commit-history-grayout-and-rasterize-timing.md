# コミット履歴グレーアウトのアルゴリズムとラスタライズタイミング

## 決定

### グレーアウトの判定ルール

右ペインのコミット履歴リストにおいて、**1つ新しい側のコミットの `changeType` が `none` または `minor` であれば、そのコミットをグレーアウト**する。

```
topItem: E (changeType=meaningful)
historyCommits = [D(none), C(meaningful), B(none), A(meaningful)]

D: topItem.changeType = meaningful → 通常表示
C: D.changeType = none             → グレーアウト
B: C.changeType = meaningful       → 通常表示
A: B.changeType = none             → グレーアウト
```

`historyCommits[0]` は `topItem.changeType`、それ以降は1つ新しい側のコミットの `changeType` を参照する。

意味：同じ視覚状態が連続しているグループの中で、**最も新しい1件だけを通常表示**し、それより古い同一状態のコミットはグレーアウトする。topItem が直前のコミットと同一状態（changeType = none/minor）の場合、historyCommits[0] もグレーアウトする。

### グレーアウトに使うデータ

`changeType`（コミット vs 親コミットのビジュアル差分）はコミット時にバックグラウンドで計算しSQLiteに保存済みの値を使う。グレーアウト表示のために追加のdiff計算は行わない。

### overlay画像の生成タイミング

赤青のoverlay差分画像は**クリック時のみ生成**する（「履歴コミット vs 現在選択コミット」の比較）。Spinnerで待機させる。履歴リスト表示時の全件先読み（prefetch）は行わない。

### ラスタライズのタイミング

| 状況 | ラスタライズ | キャッシュ |
|---|---|---|
| コミット時 | コミット対象ファイルのみ、バックグラウンドで実行 | DBに保存（blob OIDキー） |
| 過去コミット参照時 | しない | DBキャッシュを使用 |
| 未コミットファイル表示・diff | 都度ラスタライズ | しない（作業コピーは随時変化） |

## 背景

ADR-0019 では右ペインの履歴リストについて「変更していないコミットをグレーアウト」と定義したが、アルゴリズムは未定義だった。

また、ラスタライズ（PDFium, A3@400DPI）は1枚あたり数秒かかる重い処理であり、履歴リスト表示時に全コミット分を先読みすると顕著な遅延が生じていた。

## トレードオフ

### グレーアウトのアルゴリズム

**採用した理由：**
- `changeType`はコミット時に計算済みのため、履歴表示時に追加コストがゼロ
- 「変更を導入した最新の状態が見える」という直感的なUXになる

**却下した案：**
- 「各履歴コミット vs 現在選択コミット」のライブ比較 → 選択コミットが変わるたびに全件ラスタライズが必要で重すぎる
- 「チェーンをたどって現在選択コミットと同一かどうか推定する」 → 実装が複雑な割に精度が不完全（リバート等で誤判定）

**注意：** グレーアウト（vs 親）とoverlay差分（vs 選択コミット）は比較軸が異なる。グレーアウトされているコミットをクリックしてもoverlay差分が "meaningful" になることがある（例：変更後にリバートされた場合）。これは仕様。

### overlay先読みの廃止

ユーザーがクリックしない履歴アイテムのdiffを事前に生成するコストが大きく、グレーアウトもDB値で完結するため、prefetchを廃止した。クリック時のSpinner待機はDBキャッシュにより多くの場合は短時間で完了する。
